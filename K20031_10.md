## 3. Git及びGitHubに関する調査

以下の各項目について、インターネットや書籍を利用し調べ、自分なりの言葉で説明をしてください。(参考文献等あればそれも記載すること)

### 3-1. バージョン管理システムを利用することによるメリットはなにか
ファイルを編集前の状態に戻したい時、あらかじめファイルをコピーしておくのが最も手軽であるが、大きいプロジェクトの場合、自分でファイルをコピーする方法を取ると問題が発生しやすくなる。例　コピーを作りすぎてどのファイルがどのファイルかわからない。　いちいちコピーを取る手間など。
バージョン管理システムを使うことによりファイルの状態を好きなときに更新履歴として保存しておくことができる。そのため、一度編集したファイルを過去の状態に戻したり、編集箇所の差分を表示したりすることができるため個人または複数人で作業を行うときに問題やエラーが発生した場合にもすぐに対応することができる。

  ### 3-2. Gitにおけるキーワード「clone」「commit」「push」「branch」「fetch」「pull」「merge」「rebase」についてそれぞれ何をするものか (他者に説明できるよう、わかりやすく簡潔な文章で表すこと)
  
- clone
「clone」は既存の「リポジトリ」を複製します。そもそもリポジトリ（repository）とは、ソースコードや変更履歴、コメントなどを一括して保管する場所です。リポジトリには、自分のPC上に作る「ローカルリポジトリ」と、「GitHub」などのWebサービス上に作る「リモートリポジトリ」があり、両者を連携させることで複数の開発者による開発を1本にまとめることができます。既存のリポジトリ（リモートリポジトリ）にあるソースコードなどを入手したい場合は、まず、「git clone」でリポジトリを自分の環境に複製します。ローン作成を行うと、元のリポジトリを指す「origin」という名称のリモート接続が自動的に作成されます。これにより、中央リポジトリとの通信がきわめて簡単になります。
  
- commit
  そのためにファイルの変更や追加などを保存するためのコマンドである。ターミナルでcommitする場合、以下のようなオプションが存在する。
  既にgitに追加済のファイルで、変更があったものを全てコミット 　git add -a　　git commit
  または、git commit . 
  
  エディターを起動することなく、コミットメッセージも指定してコミットする git commit -m "コミットメッセージ"
  特定のファイルを指定してコミット git commit ファイル名
  
  コミットを取り消す　git reset
  
- push
  「git push」はローカルリポジトリの内容をリモートリポジトリに送信（アップロード）するコマンドまたは、リモートレポジトリのブランチ履歴を更新するための Git コマンド。
  コミットが誤って上書きされるのを防止するために、Git ではプッシュ先リポジトリにおける統合処理が早送りマージ以外である場合にはプッシュが拒否される。  
- branch
    branch は、ブランチの作成、一覧表示、名称変更、削除を行うコマンド。ブランチの切り替えを行う機能も、分岐した履歴を統合して元に戻す機能もありません。そのため、多くの場合 git branch コマンドは git checkout コマンドおよび git merge コマンドと併用されます。
  
- fetch
  fetch コマンドは、リモート リポジトリからローカル リポジトリにコミット、ファイル、参照をダウンロードします。フェッチとは、他の開発者の作業内容を確認する場合に行う操作です。コマンドは、中央リポジトリの進捗状況を確認できるという意味では svn update コマンドと似ていますが、このコマンドでは変更はリポジトリに統合されません。Git によってフェッチされたコンテンツは既存のローカル コンテンツから分離されるため、ローカルな開発作業には全く影響を与えません。
  
- pull
  pullは「トラッキングブランチの更新とローカルブランチのマージ」のことである。このとき更新されるのは、あくまで指定したリモートブランチに対応する「リモート追跡ブランチ」です。（git fetch はローカルブランチ、インデックスや作業ツリーに一切の変更を加えません。）
  
- marge
  Git において分岐した履歴を戻して統合する手段です。git merge コマンドは、git branch を使用して作成された、独立した複数の開発ラインをひとつのブランチに統合するコマンド。現在のブランチはマージの結果更新されますが、対象ブランチはそのまま残ります。従って、git merge は通常、現在のブランチを選択する git checkout および不要になった対象ブランチを削除する git branch -d と併用できる
  
- rebase
  rebase は、1 つのブランチから別のブランチへの変更を統合することを専門とした、2 つの Git ユーティリティの 1 つです。もう 1 つの変更統合ユーティリティは git merge です。マージは常に、変更レコードを先へ動かします。代わりに、rebase には強力な履歴書き換え機能を備えています。
  
  
 ### 3-3. プロジェクト進行に、GitHub Flowを採用することで解決できるチーム開発での問題はなにか (実際にGitHub Flowで作業する手順も説明する)
  GitHub Flowというのは、GitHub社が提唱している効果的なワークフローのこと。
  メリットは、簡単に理解でき、より素早く作業ができること。また何かを台無しにしてしまうとか間違ってしまった手順をやり直すといったこともめったに起こらない。プロセスを支援するためのラッパースクリプトが必要ないため、GUIプログラムも問題なく使えることがです。

  GitHub Flowの6つのルールがある。
  masterブランチは常にデプロイ可能である
  作業用ブランチをmasterから作成する（例：new-oauth2-scopes）
  作業用ブランチを定期的にプッシュする
  プルリクエストを活用する
  プルリクエストが承認されたらmasterへマージする
  masterへのマージが完了したら直ちにデプロイを行う
  
  手順は５つある。
  1. 作業ブランチの作成・切り替え
  2. ファイルの変更・追加、動作確認
  3. PullRequestの作成
  mainブランチへの取り込み準備
  4. PullRequestのレビュー依頼
  他のメンバーがレビューし、mainへ取り込み
  5. mainブランチをFetch/Pullして作業ブランチの再作成である。
  
  GitHub flow問題点は主に3つある。
  環境にどのブランチをデプロイしているの確認コースがかかる
  hotfix（不具合対応）とfeature（機能開発）が区別しないので、ブランチ運用ミスを発生する可能性が高い。
  リリースタグ切る、マージ済みブランチを削除の手順
